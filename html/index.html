Agenda

Assumptions of ideal:

  • Every service change has perfect information
  • Changes never conflict (deploy nor run)
  • Changes never fail or cause rework
  • Since changes will fail, impact is nil

Why Productivity Per Capita Suffers

Change Conflict Person-Centric Dependencies Variance in Repetitive Procedures Details Lost During Value Delivery

Change Conflicts

Dev-specific: Code (Merge): Two ppl making a chg to same file Analytical: Flow: Two chg conflct on user Flow Deploy-time: Does a whole product suite need to be brought down to make an update?
- Pair or group review, reduce parallelism in exchange for quality and speed - Componentize! - Smaller pieces are easier to modify without conflicts -- These component boundaries should usually bubble up to the analysis stage - Microservices, and pulling sub-application interfaces up (opening them to analysis) - Deploy books that allow partial application update, and even gradual/graceful update - Smaller sub-apps and version-sensitive load balancing can -- Dark deploy: no one uses -- White list: n users access -- Balance: x% users access -- Overdeploy: old and new up -- Teardown: remove old version handlers -- (Useful to imagine checkout lines?)
- During analysis, guess which pieces will change most often -- Don't tell a developer where to put physical boundaries - Pull Request - Docker - Ansible - Jenkins - Post-deploy reports to verify change success

Person-Centric Dependencies

- Analysis: Single Wringable Neck -- Wait in line - Development: Esoteric or overengineered code - Deploy: Steve is the Process
- Always design in committee - Document agressively - Change requests come with 'whys' - Pair or group programming - Juniors and new team members doing work review for vets - Automate processes
